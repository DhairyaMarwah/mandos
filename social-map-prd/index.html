Product Requirements · Mobile · Hong Kong

Mandos: the night, collapsed onto one map

A social map that folds an events app, a party app, a table-booking app, a friends app and a voucher app into a single live view of Hong Kong. This document walks every designed screen, the journeys that stitch them together, the conditional states the mockups don't show, and the scope planned but not yet drawn — so the architecture is built flexible from day one.

Prepared for  Engineering / Tech team · Source  23 designed screens · Updated  Jun 24, 2026 · Status  Design → Spec
TL;DR

Mandos is a live, dark-themed Apple-MapKit map of Hong Kong where every social primitive is a marker: events (pin size scales with popularity), users who are "out" (avatar + status pill), clubs & bars, regions (e.g. Midnight City), and "Mandos Coin" vouchers. Users filter by time (Today / This Weekend / This Week / Select Date) and category (All / Events / Friends / Clubs), tap any marker to open a detail sheet, message friends or request strangers, explore a region's people-venues-chat-offers, and claim vouchers. Two systemic threads run through everything: a visibility model (Visible ↔ Ghost, with Boost planned) and a location-permission model that degrades the whole map when off. Vouchers ship in two phases — Phase 1: tap-to-claim from anywhere; Phase 2: geofenced claim within ~100m — so proximity, anti-spoofing and per-user daily limits must be designed into the data model now.

The one rule to design around — location gates the social layer

With location OFF, Mandos still works fully as a discovery app — the user can browse all events, venues, regions and offers / vouchers on the map. What they cannot do is see people: no friends or public users appear, the user is invisible to others, and nothing is distance-ranked. The UI must state this plainly up front — "turn on location to see who's out tonight" — so the absence of people reads as a setting, not a bug, and the user knows exactly what they're missing.

With location ON, the social layer switches on: the user sees friends and public users who are "out", their live status pills and distances, and they become visible to others (unless they pick Ghost). The status pills reuse the existing status set from the current module — no new status taxonomy is introduced. Every on/off × Visible/Ghost combination is mapped in the state machine, §14.

01

The marker language

Map Icons Items

A dedicated legend screen teaches the four marker families and their visual rules. These rules are load-bearing — pin sizing, status pills and coin states all derive from data.

Events

Rounded poster thumbnails with a date badge (MAY 09). Pin size scales with popularity — needs a numeric popularityScore and a sizing function with min/max bounds, deterministic for ties, and clustering so popular events don't occlude smaller ones.

Clubs & Bars

Circular icons, constant size, showing the venue's upcoming-event poster or its profile image (precedence: upcoming event wins, fall back to profile), with a name label that must truncate.

Users

Circular avatars, constant size, with a status pill ("I'm out!"). Status text is variable → truncation + profanity/empty handling. Users in Ghost mode are excluded entirely, not drawn differently.

Vouchers / Mandos Coin

Hexagonal "M" coin with a value label ("20% discount"). Value is data-driven (%, fixed, BOGO, HKD). The legend shows only the active coin — claimed / expired / not-nearby / limit-reached states still need iconography (Phase 2 / not designed).

A Region is not a pin — it's an admin-drawn area

Unlike the four marker types above, a Region is not a point on the map. It is a category / zone created by Mandos admins from the web dashboard: an admin drops a pin and sets a radius, and that creates a region area on the map. Everything that falls inside the radius is automatically tagged to the region — the people inside, the venues inside, the offers / vouchers inside, and the users inside.

Users can see and open a region from the map. And when a user physically enters the area with location on, they get a popup notification and the region opens up to show everything inside it — who's around, the venues, the live chat box, the vouchers / offers, and so on. So the legend's four pins are the contents; the region is the container drawn around them. (Dense areas also still need cluster glyphs — M+ / +N — plus Boost/Ghost user treatments and voucher state variants.)

02

Information architecture

The map is the hub. Almost everything else is a bottom sheet that slides over it, so navigation is shallow and context is preserved when a sheet closes.

Because nearly every detail surface is a sheet over the live map, three architectural rules follow: (1) map state persists under the sheet (camera, active filter, category); (2) the sheet payload should render header content instantly from the tapped marker while the full detail loads; and (3) back / dismiss must be lossless. The one true full-screen context switch is Search.

03

Primary user journeys

8 flows

The end-to-end paths that stitch the screens together, with their decision branches. Each maps to a module spec below.

1 · First-run onboarding → live map

Onboarding

Sign-in → value/permission screen → native OS dialog → branch: Granted → live map centred on user · Approximate → map with precision warning · Denied / "keep browsing" → degraded Device Location off map → (once) marker legend → land on Core Map.

2 · Browse & open an event

Events

Select Events tab → optionally apply a time filter → scan poster pins / "Events Near you" carousel → tap → event sheet → branch: Get Directions · Send a message (inquiry/booking, auth-gated) · share / bookmark · expand for full detail.

3 · Discover a person & connect

Friends & Users

See avatars + "I'm out!" pills (and own "You" marker) → tap a user → profile sheet → branch by relationship: non-friend → Follow / request + request-to-message · friend → open existing chat. Recent-activity thumbnail → event detail.

4 · Region arrival & exploration

Regions

Pan to / tap a region or physically enter its geofence (arrival sheet) → choose: See who's around · Join Live Chat · Discover offers & venues · Explore the Zone → tabbed region view (About / People / Live Chat / Venues / Offers).

5 · Voucher claim — Phase 1 (digital)

Vouchers

Spot coin pin (value + distance) → tap → "You found a Voucher!" → Claim → server validates expiry / stock / daily limit → "Congratulations!" (added to wallet) → View all vouchers / share with friends.

6 · Voucher claim — Phase 2 (geofenced)

Phase 2

Tap coin with location ON → proximity checked vs ~100m → branch: within range → "You're within range!" claim enabled · out of range → "You're not close enough" claim blocked + Get Directions → cross geofence → sheet flips to claimable → claim (subject to daily limit / stock) → success + share.

7 · Toggle visibility (Ghost / Visible)

Visibility

Tap eye / visibility icon → "Visibility on Map" sheet → choose Visible to all (broadcast + see everyone, distance-ranked) or Ghost Mode (hidden; only see public / location-on users) → branch: Ghost+location-off shows "what still works without location" + Turn on Location deep link → mode persisted + broadcast in near-real-time.

8 · Location-off degraded recovery

Onboarding

App detects location denied / off (or a location-gated action tapped) → "You're invisible on map" reminder sheet → explains consequences + what still works (saved-venue events, friends' last spot, messaging) → branch: Open Settings / Turn on Location · Keep browsing → on return from Settings, re-check on foreground and restore full map if granted.

Module · Visibility (Ghost / Boost)

2 screens

A bottom-sheet toggle controlling both whether others see the user and what the user sees. The asymmetric rule — Ghost users only receive public / location-on users, not friends-only — must be enforced server-side in every geo/presence/search query. The four-state matrix (the core complexity)
  • Two independent variables: visibility (Visible/Ghost) × location (on/off) → four combinations, each distinct UI/copy. This screen is Ghost+Off.
  • Copy must differentiate "you chose to hide" (Ghost) from "we can't see your spot" (location off)
  • Stored visibility flag vs OS permission mismatch → backend must not advertise a user as present without a valid location

Boost mode — planned, not on this sheet

"Boost" (be more visible) is a stated feature absent from this two-option control. Model visibility as a spectrum/enum so a third elevated tier (and granular "friends-only", scheduled / auto-Ghost, timed "hide for 2 hours") can be added without redesigning the toggle.

Module · Vouchers ("Mandos Coin")

7 screens

Hexagonal "Mandos Coin" voucher pins scattered across the map with value + distance. The flagship mechanic, shipping in two phases — and the single most state-heavy module in the app. Server-authoritative limits, stock, expiry and (Phase 2) anti-spoofing are non-negotiable.

Phase 1 — Digital claim Now

Tap a coin anywhere → claim straight to the wallet. No proximity needed. Still gated by expiry, remaining stock, one-per-user, and the daily claim limit.

Phase 2 — Geofenced claim Planned

Claim only when physically within ~100m. Tap shows "nearby" vs "not nearby"; crossing the boundary flips it to claimable. Needs precise location, hysteresis at the boundary, and server-side location attestation against spoofing.

The mockups already carry the Phase-2 hooks — every voucher shows a distance ("1.2km away") and the Device Location off pill. Crucially, the same surfaces must resolve why a voucher can't be claimed through one canonical resolver rather than inferring it from scattered flags:

Location & permission

Permission & visibility state machine

The two independent variables that reshape the app: location permission (off / on) and visibility (Visible / Ghost / Boost). The diagram traces what the app does in each case — location off keeps discovery working but switches the whole people layer off; location on unlocks it, then visibility decides how the user appears. Re-evaluate the location axis on every foreground.

Location permission

  • States: not-determined · while-using · allow-once (lapses) · approximate-only · denied · restricted (MDM) · services-globally-off
  • Re-evaluate on every foreground; first-launch → native prompt, previously-denied → deep-link Settings
  • Location-off forces a ghost-like invisible state — reconcile copy with explicit Ghost mode

Visibility mode

  • Enum, not binary: visible_to_all · ghost · boost — Ghost = hidden AND only receives public/location-on users (friends-only suppressed)
  • Asymmetric Ghost view rule enforced server-side in every query
  • Block list / ACLs override visibility everywhere; presence TTL/heartbeat handles staleness

Open questions for the tech team

Ambiguities to resolve before building — each changes data shape or behaviour, so worth deciding early.

Decide before implementation
  1. Filter composition: do category tabs (All/Events/Friends/Clubs) compose with time filters as a strict AND? Are "Friends"/"out" markers time-bound or always-now?
  2. Region scope vs viewport: does the "near you" list follow the labelled region or the free-panned viewport, and which wins when they diverge?
  3. Date picker semantics: single day vs range (trigger says "Select Date", sheet says "Range")? Explicit Apply/Clear or auto-apply? Are past dates selectable? Cross-month ranges?
  4. Geofence reference point: is "range" measured to the voucher drop point or the issuing venue? Distance chips show venue distance ("1.2km") while the sheet says "within range" — needs one source of truth.
  5. Ghost in region chat: can an invisible user post, and do they appear as participants / count toward occupancy ("23")?
  6. "Friends" definition: only the viewer's friends, or all people who are "out" (friends + public)? Must be consistent across map, search, filters.
  7. Eye / region pill: does it open region detail OR toggle visibility? The same affordance is read both ways across screens — pick one canonical behaviour.
  8. Daily-limit & chat "today" timezone: Hong Kong time vs device vs server UTC, and behaviour across local midnight while travelling.
  9. Claimability copy: adopt the canonical resolver enum (CLAIMABLE / NOT_NEARBY / ALREADY_CLAIMED_BY_USER / CLAIMED_OUT / PROMOTION_ENDED / DAILY_LIMIT_REACHED) so reasons aren't overloaded.

  11. Regions & Vouchers in search: where do they live in the 4-tab search? Placeholder copy ("producers, djs & venues") understates scope.
  12. Distance when location off: hide the chip, show "approx", or use last-known/coarse/venue-cached coords? "1.2km away" currently renders even with location off.
  13. Live vs snapshot: should an open sheet update attendee counts, distance/status and voucher availability in real time, or snapshot at open?
  14. Self-broadcast contract: confirm that granting OS location does NOT imply being visible — so a Visible-but-location-off user isn't advertised without coordinates.
