The Social Map, spec'd for build

A location layer on top of Mandos: see friends who are out, events, clubs, bars and collectable rewards (coins & gifted coupons) around Hong Kong — without ever exposing anyone's exact spot. This is the concise, screen-by-screen brief for the dev team.

Feature  Social Map · Platform  Mobile (iOS shown) · Screens  13 · Status  Design → Build
TL;DR

One map shows four things: friends, events, clubs/bars, and rewards (Mandos Coins + gifted coupons). Location is opt-in and reciprocal — you see other people only while you're sharing your own location; it's sampled only while the map is open, and other users only ever see an approximate area + a "last active" time — never precise coordinates. Users can go Ghost to hide while still browsing. Rewards are coins (the currency) and coupons / bundles / passes (concrete drinks, tickets, entry) — not percent-off vouchers; admin can gift them and users buy the rest in the shop. People on the map are users who set an "Out Tonight" status; their avatar ring fades as their presence goes stale, and pins cluster when zoomed out.

01

What it is

Mandos is a social event-booking app for Hong Kong nightlife. The Social Map is the "what's happening around me right now" surface.

On a single dark map the user explores four kinds of pins — the friends they're already connected with, events happening nearby, the clubs and bars themselves, and rewards ("Mandos Coins" to collect and gifted coupons to claim). They can filter to any one type, jump to a date, open any pin for detail, and message friends. Everything sits on top of a strict privacy model: the map never reveals where a person exactly is.

Event marker size scales with popularity so hot events read first at a glance. Clubs, bars, users and reward pins are all a fixed size — a bigger pin never means a bigger venue.

03

Location & privacy — who sees what

Core rule

The most important part of this feature. Privacy is a product promise, not a setting buried in a menu.

Grant Access (first run)

Before the OS permission prompt, a priming sheet — "See who's out — share to see" — states the deal in plain language, and the deal is reciprocal: you see other people on the map only while you're sharing your own location. Turn sharing off and you go invisible — and the map hides everyone else from you too. CTA: Share my location, with an escape hatch: Keep browsing without location.

The promises shown to the user:

  • It's a two-way deal — share your location to see others; stop sharing and everyone hides from you.
  • Others only see your approximate area — a rough zone plus your last-active time, never your exact spot.
  • Location is checked only when you open the map — it updates in-app, never in the background.
  • Go invisible anytime with Ghost mode — keep browsing events & chat without sharing.
Non-negotiable privacy rules for engineering
  • Reciprocal visibility: a user receives other people's location markers only while their own location is being shared. No sharing → the people layer is hidden for them (venues, events, coins & reward drops still show). This is the core deal — "share to see".
  • Never broadcast precise coordinates of a user to other users. Coarsen server-side (rough zone / rounded geohash) before it reaches another device.
  • Other users see an approximate area label + a "last active" timestamp only (e.g. "Out around Fung Tea House · Central · Updated 12min ago"). This is the anti-stalking guarantee.
  • Foreground-only sampling: capture position when the map is opened/active. No background location tracking.
  • Location is opt-in; the whole map must remain browsable in a degraded, self-hidden mode if access is declined.

Location-off reminder

When device location is off, the top bar reads "Device Location off" and a reminder sheet — "You're invisible on map" — explains the trade-off and offers Open Settings or Keep browsing without location.

Still works without location: browse events near saved venues, and send messages / join collabs. What's lost: you don't appear to anyone, the map hides other people from you too (it's reciprocal — no sharing, no people layer), and there's no "near me" distance.

The map & filters

The default surface once location is granted.

Friends view (default map)

  • Top bar: back · current region label (e.g. "Tai Kok Tsui") with an eye/visibility icon · search.
  • Avatar row (horizontal scroll) of friends who are out.
  • Friend pins with "I'm out!" status pills; own pin is ringed and labelled You.
  • Filter chips (bottom): All Events Friends Clubs
  • "Drag Up to Explore" bottom sheet → list/region browsing.

Date filter

Filter map activity by time. Quick chips — Today This Weekend This Week Select Date — plus a calendar for a custom date range.

Dates with events near the current map view are marked in yellow, so the calendar itself hints where activity is. Selecting a date re-queries the map for that day.

05

Clustering (zoomed-out map)

Perf + legibility

When the map is zoomed out, dozens of nearby pins would overlap into noise. Instead, group them into cluster badges.

How clusters read

  • A cluster is a teardrop badge = count + a dominant-type glyph (person = people, headphones = clubs/nightlife, globe = events).
  • Mixed clusters append a small secondary icon (e.g. "12 🌐 + 👤") when a second type is significant.
  • Standout single pins stay un-clustered — a prominent event card (Justin's Event) can render on its own even when zoomed out.
  • Tap a cluster → zoom to the level where it breaks apart into individual pins.
  • Clusters respect the active filter chip — filtering to "Friends" only clusters people, etc.

When zoomed out, dense pins collapse into a single teardrop badge = count + the cluster's dominant-type glyph. The badge styles, and what a tap does:

Behaviour to build
  • Merge nearby pins into one badge as soon as they'd overlap; tap → zoom to expand.
  • Badge = count + dominant-type glyph; a mixed cluster shows a small secondary chip.
  • Rebuild on every pan / zoom, and respect the active filter chip (filtering to Friends only clusters people, etc.).
  • Standout single events can stay un-clustered so they always show.

Standard tooling handles the maths — Supercluster (react-native-maps / MapLibre / Mapbox), or native MapKit / Google ClusterManager — so this is a rendering + interaction spec, not new algorithm work.

06

People & messaging

Tapping a friend pin opens their profile sheet — the social heart of the feature.

User profile sheet

  • Approximate distance — "About 1 km away" (never an exact pin-drop).
  • Status — "Out Tonight", with an approximate area: "Out around Fung Tea House · Central".
  • Last active — "Updated 12 min ago".
  • About — bio, relationship status, home city, member-since.
  • Recent Activity — event thumbnails they've engaged with.
  • Actions: Follow and Send a message — messaging is the primary connect action.
Messaging

Users message the friends/connections they see on the map straight from the profile sheet ("Send a message"). Scope for the map release: reuse the app's existing connections + DM system; the map is a new entry point into it, not a new chat product.

Presence & last-active status

Clarified

A user avatar's ring colour encodes how fresh their presence is. The dimmer/greyer the ring, the older the "last active".

The four freshness tiers

Every marker carries a "last active" timestamp; the ring is derived from its age. Only the newest tier feels truly "live".

TierRingStatus pillAge (indicative*)
Active Green (glowing)"I'm out!"≲ 30 min
Stale Blue"I'm out!"~30–60 min
Very stale Amber"I'm out!"~1–3 hr
Inactive Grey (avatar dimmed)"Inactive"≳ 3 hr

*Thresholds in the mock (12 min / 34 min / 2 hr / 6 hr) are examples — the exact cut-offs still need to be locked. See build note.

Clarified with the team
  • How long a user stays on the map: for now, shown indefinitely — the user base is small, so we keep presence around. Later, a 2-week cap: after ~14 days of no activity a user's presence expires and drops off the map entirely.
  • Who these people are: the "random" people on the map are users who set an "Out Tonight" status — the same presence signal used on the Discover page. So the map is open discovery of who's out, not limited to your own connections.
08

Visibility — Ghost mode

A user toggle (opened from the eye icon) controlling whether they appear to others.

Visible to all

Default. Others see your approximate area + last active time — never your exact spot. You can browse events near saved venues, see friends who shared their spot, and send messages / join events.

Ghost mode

You're hidden — you won't appear to friends or show up nearby, but you can still see who's out and browse everything. Send messages and join collabs still work.

Note: Ghost mode and a device with location off both resolve to the same "hidden but fully browsing" state — see build note 2.

09

Rewards — Coins, Coupons, Bundles & Passes

Live

Mandos has one currency — Mandos Coins — and a small set of reward items: Coupons, Bundles and Passes for drinks, tickets and entry. There is no percent-off "voucher": a coupon grants a concrete thing (a free drink, an entry, buy-1-get-1), not "20% off". True percent-discount deals scoped to a venue are a future merchant feature, not something the map drops today.

Interactive mockups

See the updated screens rendered live in social-map/ — the Reward Found sheet, the Rewards shop (Coupon / Bundle / Pass with dual HK$ + Coins pricing), and the reciprocal location sheets. These replace the old voucher-found / map-vouchers screens.

On the map — two kinds of drop, both authored by Mandos

  • Mandos Coins — the hexagonal "M" coin, labelled with a coin value (+50), not a discount. Tap to collect it straight into your balance.
  • Gifted rewards — a gift pin (e.g. a free-drink coupon) that Mandos hands out. Tap to open the Reward Found sheet and add it to My rewards.
  • Both are filterable like any other pin. (No "20% OFF · 2km" pins — that was the old voucher model.)

Reward Found sheet

  • Type — COUPON / BUNDLE / PASS chip, plus what it grants ("1× Drink", "3× Drink + 3× Ticket", "1× Entry").
  • Source — "Gifted by Mandos" (map drops are admin-gifted; the paid catalogue is bought in the shop).
  • Venue & where it dropped — e.g. "Boomerang Club · Central".
  • Terms — numbered (one per user, valid-till date, not on special-event nights, not combinable).
  • Actions: Send as a gift and Add to my rewards. Owned coupons live in My rewards and redeem by QR at the venue.

Rewards shop

  • Two tabs — Shop (buy) and My rewards (owned).
  • Every item carries a dual price — HK$ and Coins — and a category (Bundles / Tickets / Drinks). Examples: Coupon "Zonal Drink · 1× Drink", Bundle "3× Drink and 3× Ticket", Entry "Boomerang Ticket · 1× Ticket".
  • Admin-gifted items show "Gifted · Free" and skip payment; users buy the rest with HK$ or redeem with Coins.
⚑ Dev note — model & authoring
  • Coins are the only currency. Rewards are coupons / bundles / passes, each granting a concrete thing (N drinks / tickets / entry) — never a % discount.
  • Authoring: Mandos admin creates coupons & bundles in the dashboard and can gift them free (free-drink coupon, free-drink pass, coins) to users. Merchant-level percent deals are a separate, later feature.
  • States: available → owned (My rewards) → redeemed (QR at venue); validity window; "not combinable"; and gift transfer to another user.
  • Phase 2 (later): geofenced drops — a gifted reward only claimable near the venue/region. Model it now so the proximity gate switches on without a rewrite.

Build notes & open questions

The things worth agreeing on before code.

Decided / clarified
  1. Coarsen every user location server-side; broadcast area label + last-active time only.
  2. Foreground-only location, sampled on map-open. No background tracking.
  3. Only event markers scale by popularity; all others constant.
  4. Presence retention: shown indefinitely for now (small user base); 2-week cap later — no activity in ~14 days → drop off the map.
  5. Map population = "Out Tonight" users — same signal as the Discover page; open discovery, not just your connections.
  6. Cluster markers when zoomed out (Supercluster or native), count + dominant-type glyph, tap-to-expand.
  7. Rewards: coins & gifted coupons dropped on the map now, geofenced later; coupons/bundles authored via web dashboard.
  8. Map must stay browsable with location denied (self-hidden).
Open questions for the team
  1. Last-active thresholds — lock the exact cut-offs for Active / Stale / Very-stale / Inactive rings (mock shows 12 min / 34 min / 2 hr / 6 hr as examples).
  2. Grant-access timing — confirm the priming screen shows on first Social Map open (before the OS prompt), and how often the location-off reminder re-surfaces after that.
  3. Distance granularity — snap "km away" to buckets (e.g. 500 m / 1 km)? What's the coarsening radius for the area label?
  4. Ghost vs location-off — both hide the user; confirm whether Ghost still requires location ON (to power "near me"), or is purely a broadcast toggle.
  5. Reward gifting — how does "Send as a gift" transfer ownership, and does the recipient inherit the same validity/geofence?
  6. Claim limits — any per-user daily cap on coin collection / coupon claims?
